- 登录校验：
	- 概述：是指系统在用户登录时，验证其提交的身份凭证（如用户名/密码、手机号/验证码等）是否合法有效的过程。
	- 会话技术：
		- 概述：用户打开浏览器，访问 web 服务器的资源，会话建立，直到有一方断开连接，会话结束，在一次会话中可以有多次请求和响应
		- 会话跟踪：
			- 概述：一种维护浏览器状态的方法，服务器需要识别多次请求是否来自同一个浏览器，以便在同一会话的多次请求中共享数据
			- 方案：
				- 客户端会话跟踪技术：Cookie
					- 概述：是服务器发送到用户浏览器并保存在本地的一小段数据（通常不超过 4KB），浏览器会在后续访问同一服务器时自动携带这些数据，从而让服务器能够识别用户、维持状态或记录信息。
					- 工作原理：
						1. **设置 Cookie**：服务器在处理请求时，通过 HTTP 响应头 `Set-Cookie` 告知浏览器保存一个 Cookie（包含名称、值、过期时间等属性）。
						2. **存储 Cookie**：浏览器根据指示将 Cookie 保存在本地（内存或磁盘中）。
						3. **发送 Cookie**：之后每次向同一域名发起请求，浏览器都会自动在请求头 `Cookie` 中携带所有符合条件的 Cookie，服务器据此识别用户。
					- 优缺点：
						- http 支持的技术。
						- 移动端无法使用 cookie
						- 不安全，用户可以禁止 cookie
						- cookie 不能跨域
				- 服务端会话跟踪技术：Session
					- 概述：Session 通过在服务器端保存用户上下文，让应用能够“记住”用户
					- 工作原理：
						1. **创建 Session**  当用户首次访问应用时，服务器检查请求中是否携带 Session ID。如果没有，则创建一个新的 Session 对象，并生成唯一的 Session ID（例如 32 位随机字符串）。
						2. **传递 Session ID 给客户端**  服务器将 Session ID 通过 **Cookie**（默认方式）发送给浏览器，Cookie 中通常包含 `JSESSIONID`（Java 中的默认名称）和路径等信息。
						3. **客户端携带 Session ID 访问**  浏览器后续请求会自动在 Cookie 中带上 Session ID，服务器根据 ID 查找对应的 Session 对象。
						4. **读写 Session 数据**  服务器在处理请求时，可以从 Session 中读取之前存储的数据，也可以添加新数据。这些数据保存在服务器内存（或其他存储）中，不会暴露给客户端。
					- 优缺点：
						- 储存在服务端，安全
						- 服务器集群情况下无法使用 Session
						- 与 Cookie 相同的缺点
- JWT 令牌技术：
	- 概述：全称 Json Web Token，定义了一种简单，自包含的格式，用于在通信双方以 json 数据格式安全传输信息。由于数字签名存在这些信息是可靠的
	- 组成：
		- 第一部分：头部（header），记录令牌类型，签名算法等
		- 第二部分：有效载荷（Payload），携带自定义信息，默认信息等
		- 第三部分：签名（Signature），防止 Token 被篡改确保安全性。将头部、有效载荷加入指定密钥，通过指定签名算法计算而来
	- 工作原理：
		1. **用户登录**：客户端（如浏览器、移动 App）提交用户名和密码到服务器。
		2. **生成 JWT**：服务器验证凭据成功后，创建一个 JWT，将用户标识（如用户 ID）、角色、过期时间等信息放入 Payload，并使用密钥签名。
		3. **返回 JWT**：服务器将 JWT 返回给客户端（通常放在响应体或 HTTP 头中）。
		4. **客户端存储 JWT**：客户端（如浏览器可以使用 localStorage 或 sessionStorage，移动端可以安全存储）保存该令牌。
		5. **携带 JWT 访问**：后续每次请求，客户端在 HTTP 请求头中加上 `Authorization: Bearer <token>`。
		6. **验证 JWT**：服务器收到请求后，从请求头中提取 JWT，验证签名是否有效、是否过期，然后解析出用户信息，决定是否允许访问。

- Filter：
	- 概述：Filter 过滤器，是 JavaWeb 三大组件之一，可以把对资源的请求拦截下来，从而实现登录校验，统一编码，敏感字符处理等特殊功能
	- 实现：
		- 在类前添加 `@WebFilter("/*")` 注解，注解的值是拦截范围，有三种：
			- 拦截具体路径： `@WebFilter("/login")` 只有访问 login 这一路径时才会被拦截
			- 目录拦截：`@WebFilter("/login/*")` 访问 login 及其所有子域名都会被拦截
			- 全部拦截： `@WebFilter("/*")` 访问所有路径都会被拦截
		- 需要继承 javax.Servlet.Filter 接口，重写三个方法，init、doFilter、destroy，init 和 destroy 方法不常用，现在可以不进行重写，在 doFilter 中书写对拦截到的请求的处理
		- 还需要在启动类的上方添加 @ServletComponentScan 注解，开启对 Servlect 组件的支持
		- 示例：![[Filter简单示例.png]]
		- 执行流程：
			- 首先被过滤器拦截，然后在过滤器中执行操作，放行后进入服务器执行，然后从服务器中返回，再被拦截，执行放行后操作（这部分操作写在 doFilter 后），最后才会返回到服务器
		- 过滤器链：
			- 概述：在一个后端服务器中，可以配置多个过滤器，这些过滤器组合在一起就组成了过滤器链，进入时需要从外到内一个个放行，出来时需要从内到外一个个放行。
			- 排序：注解配置的过滤器，按照过滤器名称从 A-Z 进行排序，从 A 执行到 Z
