- # 阻塞队列的问题

	- JVM 内存有限，在高并发情况下，容易出现内存不足的问题
	
	- JVM 内存没有持久化，宕机之后所有数据都会消失


- # 消息队列介绍

	- 消息队列（message queue）就是存放消息的队列，最简单的消息队列包括三个角色：
	
		- 生产者 ： 发送消息到消息队列
		  
		- 消息队列 ： 储存和管理消息，也被称为消息代理
		  
		- 消费者 ： 从消息队列获取消息并处理消息


- # 基于 List 实现消息队列

	- Redis 中 List 数据是一个双向链表，很容易模拟出队列效果
	
	- 队列出口和入坑不在同一边，因此可以使用 Lpush 和 BRpop 或者 BLpop 和 Rpush 实现，正常的 Rpop/Lpop 当列表中没有元素时会返回 null，因此需要使用 BRpop/BLpop，在列表中没有数据时阻塞等待，直到有数据（pop是删除并返回最后一个元素）
	
	- ## 优缺点
	  
		- ### 优点：
		
			- 利用 Redis 储存，不受 JVM 内存上限限制
			
			- 利用 Redis 持久化机制，数据安全有保证
			
			- 可以满足消息的有序性
		
		- ### 缺点：
		
			- 无法避免消息丢失，从消息队列中取到消息，但还没处理就宕机时数据就会丢失，该现象无法避免
			
			- 只支持单消费者，一条消息只有一个消费者可以使用


- # 基于 PubSub 实现消息队列

	- ## PubSub 简介
	
		- PubSub（发布订阅）是Redis2.0 引入的消息传递模型，顾名思义，消费者可以订阅一个或多个 channel，生产者向对应 channel 发生消息后，所有订阅者都能收到相关消息，常用命令：
		
			- `subscribe channel [channel]` 订阅一个或多个频道
			
			- `publish channel msg` 向一个频道发生消息
			
			- `psubscribe pattern [pattern]` 订阅所有与 pattern 格式匹配的频道
				
				- 该命令可以使用通配符，如下：
				
					- `he?ll`： ？表示匹配一个字符
					
					- `he*ll`：`*`表示匹配0个或多个字符
					  
					- `h[ae]ll`：`[]` 表示可以匹配其中的任意一个字符，比如 hall/hell
	
	- ## 优缺点：
	
		- ### 优点：
		
			- 采用发布订阅模型支持存在多个生产者和多个消费者
		
		- ### 缺点：
		
			- 不支持数据持久化
			
			- 无法避免数据丢失
			
			- 消息堆积有上限，超出时数据丢失


- # 基于 Stream 的消息队列

	- ## Stream 简介
	
		- Stream 是 Redis5.0 引入的一种新的数据类型，专门用来实现消息队列，核心函数有两个。`xadd` 向消息队列中添加数据，`xread` 读取消息队列中的数据
		
		- xdd 命令参数![[xadd命令参数.png|10000000]]
		
		- xread 命令参数 ![[xread命令参数.png|1000000]]
			- block 参数中，0 是无限阻塞，直到收到消息
			  
			- 当指定起始 id 为 $ 时，在处理消息的过程中，来了两条及以上的消息，也只能获取一条，会出现漏读
			
			- ### 特点：
			
				- 消息可回溯
				  
				- 一个消息可以被多个消费者读取
				  
				- 可阻塞读取
				
				- 有消息漏读风险
