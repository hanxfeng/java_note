- # 商户查询缓存

	- ## 什么是缓存
	
		- 缓存是数据交换的缓冲区，是储存数据的临时地方，读写性能较高
	
	- ## 缓存的作用与成本
	
		- ### 作用
		
			- 降低后端负载
			  
			- 提高读写效率，降低响应时间
		
		- ### 成本
		
			- 数据一致性成本
			  
			- 代码维护成本
			  
			- 运维成本
	
	- # 添加缓存流程
	
		1. 从 Redis 中查询数据
		   
		2. 判断 Redis 中是否存在查询的数据
		   
		3. 如果存在数据，则返回查询到的数据
		   
		4. 如果不存在数据，则到数据库中查询数据
		   
		5. 如果数据库中不存在数据则返回错误，如果数据库中存在数据，则先将数据储存到 Redis 然后再返回
		
		6. 如果需要对数据库中的数据进行更新，先修改数据库中的数据，再删除缓冲，这样比先删除缓存再修改数据库中的数据出现数据不一致的可能性要低
		
			-  因为是修改数据库和删除缓存是两步执行，因此中间会存在间隙，由于 cpu 执行线程的随机性，此时可能有其他线程进行查询操作，先删除缓存，查询会直接查数据库中的数据，但此时数据库中的数据还没有更新，就会出现数据不一致。先修改数据库，此时缓存未删除，查到的就是缓存中的数据，因为缓存删除时间远高于数据库更新，因此先修改数据库出现数据不一致的可能更小
		
	
	- ## 缓存更新策略  

|      | 内存淘汰                                            | 超时剔除                                 | 主动更新                                          |
| :--- | :---------------------------------------------- | :----------------------------------- | :-------------------------------------------- |
| 说明   | 不用自己维护，利用Redis的内存淘汰机制，当内存不足时，自动淘汰部分数据，下次查询时更新缓存 | 给换缓存的数据加上TTL生存时间，到时间后自动销毁，下次查询时再进行缓存 | 编写业务逻辑，在对数据库中的数据进行修改时，同时更新缓存中的数据。要先修改数据库再删除缓存 |
| 一致性  | 差，只有当内存不足时才会更新数据                                | 一般，数据更新速度高于内存淘汰，但仍会有数据不一致的情况出现       | 好，几乎不会出现数据库数据与缓存数据不一致的情况                      |
| 维护成本 | 无，不许要进行额外处理，Redis会自动进行                          | 低，在写入缓存时加上自动删除时间即可，额外处理操作很少          | 高，需要自行编写相关代码                                  |
		业务场景：
			长时间不会更新、一致性需求较低的数据选择内存淘汰机制，节省维护成本。经常更新、一致性较高的数据使用主动更新机制，并以超时剔除方案作为兜底，防止数据不一致


- # 缓存可能出现的问题

	- ## 缓存穿透
	
		- ### 概述：
		
			- 缓存穿透是指客户端请求的数据在缓存和数据库中都不存在，这样缓存永远不会生效，这些请求都会打到数据库，攻击者可以使用多个 IP 去请求空数据，达到攻击数据库的目的
			
		- ### 解决方案
		
			- #### 1.缓存空对象
			
				- 当请求了一个缓存和数据库都没有的空对象时，在缓存中添加一个空对象，下次查询时查询到的就是这个空对象
				
				- 优点：实现简单，维护方便
				  
				- 缺点：会产生额外的内存消耗，每当出现一个空对象，都要将其添加到缓存中，消耗缓存的内存
				  可能造成数据不一致，当请求后向数据库添加了这个数据，后续再请求依然会获得缓存中的空对象
				  有限解决方案：为缓存中的空对象添加 TTL 生存时间 | 在添加数据时手动替换缓存中的数据
			
			- ### 2.布隆过滤
			
				- 添加一个布隆过滤器，它可以在不访问数据库的前提下判断数据是否存在于数据库（原理学到再详细写，判断存在误差，判定为不存在的数据几乎不会判断错误，判定为存在的数据可能不存在），如果存在则放行，不存在则拒绝访问
				
				- 优点：内存占用少
				  
				- 缺点：实现复杂，存在误判可能  
	
	- ## 缓存雪崩
	  
		- ## 概述
		  
			- 缓存雪崩是指同一时间大量缓存数据失效（TTL生存时间归零 ）或 Redis 服务器宕机，导致大量请求到达数据库，带来巨大压力
		
		- ## 解决方法
		
			- 给不同的 key 添加随机的 TTL 生存时间
			  
			- 利用 Redis 集群提高服务的可用性
			  
			- 给缓存业务添加降级限流策略
			 
			- 给业务添加多级缓存
			  
	- ## 缓存击穿
	
		- ### 概述
		  
			- 缓存击穿问题也叫热点 Key 问题，就是一个**高并发**并且**请求比较复杂缓存重建困难**的 key 突然失效了（TTL 过期或内存淘汰），每次请求都会访问数据库并尝试重建缓存，导致数据库受到冲击的现象
		
		- ### 解决方法
		
			- #### 互斥锁
			  
				- 在缓存失效后，第一个请求的线程会获得一个互斥锁，之后由该线程负责缓存重建工作，其余线程在获取互斥锁时失败，会不断重复查询缓存，获取互斥锁失败的过程直到缓存重建成功
				  
				- 优点：没有额外的内存消耗，能保证数据的一致性实现简单
				  
				- 缺点：只有一个线程进行缓存重建，其余线程只能等待，性能受影响。困难产生死锁风险
				
				- ![[缓存击穿-互斥锁.png]]

			- #### 逻辑过期
			  
				- 逻辑过期是指，不给数据设定 TTL ，而是在数据中加入一个类似 timeout 的部分，里面记录的是数据的过期时间，获取到数据后，要对比当前时间和数据中时间，确认是否过期，如果过期，则获取互斥锁，然后开启一个新线程进行缓存重建，重置过期时间，旧线程则返回过期数据，有其他线程查询时，发现数据过期后，会尝试获取互斥锁，获取失败则返回过期数据
				  
				- 优点：线程无需等待，性能良好
				  
				- 缺点：不能保证数据一致性，有额外内存消耗，实现复杂

- # 秒杀业务

	- ## 全局唯一id生成器
	
		- 全局 id 生成器是一种在分布式系统下用来生成全局唯一 id 的工具，一般满足以下特性：唯一性、高可用、高性能、递增性（可不是连续递增），安全性（不能有明显规律性）。可以通过 Redis 实现
		
		- 为了增加 id 的安全性，可以不直接使用 Redis 自增的数值，而是拼接一些其他信息，拼接id的组成部分：
		
			- 符号位，1bit，永远是0。
			
			- 时间戳，31bit，以秒为单位
			  
			- 序列号，32bit，秒以内的计数器，支持每秒产生2^32 个不同id
		
		- ## 示例
		  
		```java
		@Component
		public class RedisIdWorker () {
		
			public Long nextId(String keyPrefix) {
				// 生成时间戳
				LocalDateTime now = LocalDateTime.now();
				
				// 获取当前秒数
				Long nowSecond = now.toEpochSecond(ZoneOffset.UTC);
				// BIGIN_TIMES 是一个手动设定的 LocalDateTime 类型的数据
				Long timetemp = nowSecond - BIGIN_TIMES;
				// 生成序列号
				// 获取当前日期，单位天
				String date = now.format(DatetimeFormatter.ofPattern("yyyymmdd"));
				// 拼接获取序列号
				Long count = stringRedisTemplate.opsForValue()
					.increment("icr"+keyPrefix+date);
				
				// 先利用位运算使时间左移32位，此时，空余的32位全是0,然后使用或，count与
				// timestamp 中都是0则果是0,有一个1则结果是1,这样就实现了拼接
				return timestamp << 32 || count;
			}
		}
		```
	
	- ## 超卖问题
	
		- 在进行一些比如秒杀或限时业务时，可能会出现超卖问题，比如开始计划卖出一百份，但实际却卖出了110份。这是由程序设计和cpu线程执行特点共同引起的高并发情况下出现的问题，假设业务流程为：用户点击购买 -> 查询数据库中是否有剩余 -> 有则进行购买并在数据库中扣减 ， 无则返回错误信息 -> 用户付款，业务结束。在查询和修改之间会有时间差，假设还剩最后一份，a先进行查询，发现有剩余后继续进行，在扣减执行之前，另一个线程的b也进行查询，因为此时还没有扣减，所以b返回的结果也是有剩余。这就是超卖问题
		
		- ### 解决方法
		  
			- #### 悲观锁
			  
				- 认为线程安全问题一定会发生，因此在操作数据之前先获得锁，确保线程串行执行，因为效率偏低，且线程安全问题发生概率较低，因此更多使用乐观锁。
			
			- #### 乐观锁
			  
				- 认为线程安全问题不一定会发生，因此不加锁，只是在更新数据时判断有没有其他线程对数据做了修改。如果没有修改则认为安全，此时进行数据更新，如果进行了修改则说明发生了安全问题，此时进行重试或报错
			
			- #### 乐观锁的实现方式
			  
				- 单独设置一个版本号，每次对操作数据时，使版本号的值+1,在更新数据之前检查查询时的版本号和更新时的版本号是否相同，相同则更新，不同则报错。
				
				- 可以将剩余库存数作为版本号来简化操作
				  
				- 严格乐观锁会造成购买失败率过高的现象，因为只要剩余库存数对不上就会失败，即使库存中还有剩余，因此可以将检查条件改为检查库存数是否大于0而不是是否与查询数据时相同
