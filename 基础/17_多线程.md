- # 概述：
	- 线程：操作系统能够进行运算调度的最小单位，被包含在进程中，是进程中的实际运行单位
	
	- 进程：程序的基本执行实体
	
	- 并发：同一时刻，有多个指令在单个 cpu 上交替执行
	
	- 并行：同一时刻，有多个指令在多个 cpu 上同时执行


- # 实现方式：
	1. 继承 Thread 类的方式实现
		1. 在创建类 MyThread 时继承 Thread 类
		
		2. 在 MyThread 中重写 run 方法
		
		3. 创建 MyThread 类的对象 t
		
		4. 使用 `t.start()` 启动线程，多线程就创建多个对象然后调用
		- ![[多线程第一种实现方法.png]]
	
	2. 实现 Runnable 接口的方式实现
		1. 创建类 MyRunnable 实现 Runnable 接口
		
		2. 在类中重写 run 方法
		
		3. 创建 MyRunnable 的对象 r 和 Thread 的对象 t
		
		4. 在创建 Thread 的对象 t 时将 r 传入
		
		5. 调用`t.start()` 方法
		- ![[多线程实现第二种方法.png]]
	
	3. 利用 Callable 和 Future 接口实现
		1. 创建一个类 MyCallabe 实现 Callable 接口
		
		2. 在类中重写 call 方法，有返回值
		
		3. 创建 MyCallabe 的对象 mc
		
		4. 创建 FutureTask 的对象 ft，`FutureTask<call方法的返回值> ft = new  FutureTask<>(mc);`
		
		5. 创建 Thread 的对象 t ，传入 ft
		
		6. 使用 `t.start();` 启动线程
		
		7. 获取多线程运行结果，`返回值类型 result = ft.get();`
		- ![[多线程实现第二种方法.png]]


- # 常用实现方法：
	- ![[多线程常见成员方法.png]]
	
	- 未设置线程名称时，线程有默认名称，格式为 Thread-x ，x 是从 0 开始的序号
	
	- 需要给线程设置名字时，可以用 setName 方法设置，也可以在类中写构造方法，传入 String 类型的数据即可
	
	- currentThread 方法和 sleep 方法都是哪个线程执行到这个方法，哪个线程执行这个方法
	
	- 抢占式调度与非抢占式调度：
		- 线程的调度有两种方法，抢占式调度和非抢占式调度，抢占式调度是指哪条线程抢占到 cpu，哪条线程执行，具有随机性，非抢占式调度指所有线程轮流执行，java 中使用抢占式调度。
		
		- java 中优先级从 1-10 ，默认优先级是 5，越大抢占到 cpu 的概率越大
	
	- 守护线程：
		- 当其他非守护线程结束时，守护线程会陆续结束（不是一次性结束）
	
	- 礼让线程/出让线程：
		- 在执行到这个方法时，让出 cpu，一般写在线程中
	
	- 插入线程：
		- 将调用这个方法的线程插入到当前线程前先执行


- # 线程的状态：
	- 线程的生命周期可以概况为以下四步：
		1. 新建：创建线程对象
		
		2. 就绪：程序已经就绪，处在抢 cpu 状态
		
		3. 死亡：线程执行结束，被销毁
		
		4. 等待：使用 wait 方法使程序进入不执行的等待状态，直到被唤醒
		
		5. 阻塞：因为种种原因无法获取到锁而无法执行程序的状态
		
		6. 计时等待：使用 sleep 方法而暂时停止执行，到时间后会继续执行


- # 同步代码块：
	- ## 概述：
		- 把操作共享数据的代码锁起来
	- ## 格式：
		- `synchronized(锁){操作共享数据的代码}`
		
		- 锁默认打开，有一个线程进去，锁自动关闭
		
		- 锁中所有代码执行完毕，线程出来，锁自动打开
		
		- 锁对象一定是唯一的，即需要在前面加 static 关键字
		
		- 锁对象不一定要在操作共享数据的代码中出现


- # 同步方法：
	- ## 概述：
		- 与同步代码块类似，同步方法是将整个方法都锁起来，在方法前加 synchronized 关键字即可
		
		- 同步方法的锁对象不能自己指定，由自己进行指定，非静态方法是 this，静态方法是当前类的字节码文件对象


- # lock 锁：
	- ## 概述：
		- lock 锁可以显式的获取和释放锁
	
	- ## 使用：
		- Lock 是一个接口，不能直接创建它的对象，一般是创建它的实现类 ReentrantLock 的对象：`Lock lock = new ReentrantLock();`
		
		- 加锁：`lock.lock();`
		
		- 释放锁：`lock.unlock();`
		
		- 如果需要多次创建锁所在类的对象，需要在创建时加上 static 关键字：`static Lock lock = new ReentrantLock();`


- # 死锁：
	- ## 概述：
		- 指两个或多个线程在争夺共享资源时，每个线程都在等待其他线程释放锁，导致所有线程都无法继续执行，永久阻塞的状态。
	
	- 经典示例：
```java
			// 线程1
			synchronized (lockA) {
			    Thread.sleep(100); // 确保线程2获得lockB
			    synchronized (lockB) {
		        // 执行操作
			    }
			}

			// 线程2
			synchronized (lockB) {
			    Thread.sleep(100); // 确保线程1获得lockA
			    synchronized (lockA) {
			        // 执行操作
			    }
			}
```


- # 等待唤醒机制
	1. 三个核心方法
		- 前提：必须在同步代码块或同步方法中，通过**锁对象**调用
		
	    - `void wait()`：
	        - 使当前线程进入等待状态（阻塞），并**释放锁**
	        
	        - 直到其他线程调用 `notify()` 或 `notifyAll()` 唤醒，或线程被中断
	        
	        - 若在非同步环境中调用，抛出 `IllegalMonitorStateException`
	    
	    - `void notify()`：
	        - 随机唤醒一个正在等待该锁对象的线程（由 JVM 调度，不一定是“公平”唤醒）
	        
	        - 被唤醒的线程需要重新竞争锁，获得锁后才能继续执行
	    
	    - `void notifyAll()`：
	        - 唤醒所有等待该锁对象的线程
	        
	        - 所有线程一起竞争锁，但只有一个能获得锁并执行，其余继续等待
	
	 2. 阻塞队列方式实现
		- 概述：阻塞队列（`BlockingQueue`）内部自动处理线程安全，当队列满时，添加操作会阻塞；当队列空时，获取操作会阻塞。它简化了生产者-消费者模型的实现。
		
		- 继承接口层次：
		    - `Iterable` 接口：支持迭代器与增强 for 遍历
		    
		    - `Collection` 接口：单列集合的基础接口
		    
		    - `Queue` 接口：队列的基本操作（入队、出队等）
		    
		    - `BlockingQueue` 接口：阻塞队列核心接口，扩展了可阻塞的入队/出队方法
		
		- 常用实现类：
		    - `ArrayBlockingQueue`：
		        - 底层为数组，**有界**队列，创建时必须指定容量
		        
		        - 内部已实现线程安全（基于 ReentrantLock），使用时无需额外同步
		        
		        - 创建：`ArrayBlockingQueue<数据类型> queue = new ArrayBlockingQueue<>(容量)`
		        
		        - 常用方法：
		            - `queue.put(E e)`：向队尾添加元素，若队列满则阻塞
		            
		            - `E 变量名 = queue.take()`：从队首取出元素，若队列空则阻塞
		    
		    - **`LinkedBlockingQueue`**：
		        - 底层为链表，可选有界队列，默认容量为 `Integer.MAX_VALUE`（可视为无界，但实际受内存限制）
		        
		        - 创建时可指定容量：`new LinkedBlockingQueue<>(容量)`，若不指定则为无界
		        
		        - 同样线程安全，性能通常高于 `ArrayBlockingQueue`（链表实现，入队出队锁分离）
		        
		        - 常用方法与 `ArrayBlockingQueue` 相同
	
	 - 使用示例：
```java
			// 生产者线程
			BlockingQueue<String> queue = new ArrayBlockingQueue<>(10);
			queue.put("数据");  // 若队列满则阻塞
			// 消费者线程
			String data = queue.take();  // 若队列空则阻塞
```
	优点：无需手动 `wait/notify`，代码更简洁安全，且支持超时、中断等高级功能。


- # 线程池：
	- ## 核心原理：
		1. 创建一个池子，初始是空的
		
		2. 提交任务时，池子会创建新的线程，任务执行完毕，线程还给池子，下次提交任务时，不需要创建新线程，复用已有线程即可
		
		3. 任务提交时，池中没有空闲线程，也无法创造新的线程，任务就会排队等待
	
	- ## 代码实现：
		- ### 创建线程池：
			- `ExecutorService pool = Executor.newCachedThreadPool()`
				- 创建一个无上限的线程池
			
			- `ExecutorService pool = Executor.newFixedThreadPool(int nThreads)`
				- 创建一个上限为 nThreads 的线程池
		
		- ### 提交任务：
			- `pool.submit();`
				- 可以传入 Runnable 的实现类或 Callable 的实现类
		
		- ### 关闭线程池：
			- `pool.shutdown()`
	
	- ## 自定义线程池：
		- ### 核心参数：
			- 核心线程数量
				- int 类型
			
			- 最大线程数
				- 核心线程+临时线程数，int 类型
			
			- 空闲时间（值）
				- 临时线程空闲多久会被销毁
			
			- 空闲时间（单位，秒等）
				- 临时线程空闲多久会被销毁，值+单位，使用 TimeUnit 指定
			
			- 阻塞队列
				- 传入阻塞队列两种常用实现类的对象均可
			
			- 创建线程方式
				- 传入 `Executors.defaultThreadFactory()`
			
			- 要执行任务过多时解决方式
				- 当提交任务>核心线程+临时线程+队伍长度时触发
				
				- 处理策略
					- `AbortPolicy`：默认策略，丢弃任务并抛出异常
					
					- `DiscardPolicy` ：丢弃任务，但不报异常，不推荐
					  
					- `DiscardOldestPolicy`：抛弃任务中等待最久的任务，然后把当前任务加入队列中
					  
					- `CallerRunsPolicy`：调用任务的 run 方法绕过线程池执行
					  
				- 传入：`new ThreadPoolExecutor.上述处理方式之一`
				  
			- 线程池中有核心线程和临时线程，当提交任务时，核心线程处理任务，当任务过多时，进行排队，如果排队长度大于指定值，就会使用临时线程去处理后续任务，因此不是先提交就一定先执行
				- ![[线程池.png]]
		- 创建示例：
			- ```java
			  ThreadPoolExecutor pool = new ThreadPoolExecutor(
				  3, // 核心线程数
				  6, // 最大线程数
				  60, // 空闲时间（值）
				  TimeUnit.SECONDS, // 空闲时间（单位）
				  new ArrayBlockingQueue<>(3), // 阻塞队列
				  Executors.defaultThreadFactory(), // 创建线程方式
				  new ThreadPoolExecutor.AbortPolicy() // 任务拒绝策略
			  );
			  ```


- # 杂七杂八：
	- ## 最大线程数：
		- 与 cpu 有关，我的电脑是 14 核 20 线程，最大并行数为 20
	  
	- ## 确定线程池大小：
		- ### cpu 密集型：
			- 主要用于计算，对本地文件或数据库读写较少
			  
			- 线程池大小：最大并行数+1
		
		- ### I/O密集型：
			- 主要对本地或数据库文件进行读写
			  
			- $最大并行数*期望cpu利用率*总时间（cpu计算时间+等待时间）/cpu计算时间$
